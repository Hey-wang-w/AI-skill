# AI知识学习记忆系统 Skill

> 本Skill是管理AI知识学习、出题测验、批改复习、知识点入库的唯一执行依据。
> ⚠️ 所有规则必须严格遵守，不得自行简化或跳过步骤。

---

## 一、术语前置解释（零基础必读）

| 术语 | 解释 |
|------|------|
| 知识点（KP） | Knowledge Point，飞书表格里一条独立的知识卡片，有唯一ID |
| 艾宾浩斯遗忘曲线 | 记忆规律：学完后遗忘先快后慢，按"添加后第1/3/7/15/30天"累计间隔复习形成长期记忆；答错则归零重走（脚本自动计算，AI无需手动算日期） |
| 薄弱点 | 答错的知识点，会被高频考察直到连续2次到期答对 |
| 薄弱点到期 | 来源标签之一：该知识点已到艾宾浩斯复习日期，且该知识点被标记为薄弱点（之前答错过），需标🔥 |
| 到期常规 | 来源标签之一：该知识点已到艾宾浩斯复习日期，但不是薄弱点（正常复习），不标🔥 |
| 未到期巩固 | 来源标签之一：该知识点尚未到期，但因题量不足被提前抽中巩固，若该知识点是薄弱点则标🔥 |
| 新知识点 | 来源标签之一：刚入库、第一次被考到的知识点，不标🔥 |
| 随机巩固 | 来源标签之一：从已学过的知识点中随机抽取巩固，不推进轮次（FIX#4保护），不标🔥 |
| 正式测验 | 更新飞书复习记录的测验；纯练习不改动飞书 |
| ★★★/★★☆/★☆☆ | 知识点重要性三级：核心/重要/补充，决定出题频率 |
| FIX#1~#7 | 7个历史逻辑Bug修复的保护规则，脚本中已实现，批改时需注意理解规则即可 |

## 三、流程A：出题测验（高频核心流程）

### 执行步骤（必须严格按顺序执行，不得跳步）

**步骤A1：执行拉题脚本**
- ⚠️ 强制运行脚本，不得跳过：
  ```powershell
  python d:\AI\AI学习内容记忆测试\ai-quiz-system\scripts\quiz_pull.py
  ```
- 脚本会自动从飞书拉取知识点、按优先级筛选、生成`today_quiz.json`和出题指令。
- 等待脚本执行完成，确认输出"出题指令已保存"。

**步骤A2：读取格式规范+经验预警**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\01-question-format.md`
- 重点关注：填空题的下划线格式要求、选择题选项规则、专业名词解释要求。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\risk-alerts.md`
- 出题前预警：检查是否有与本次出题相关的已知风险。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\quiz-experience.md`
- 出题时参考：检查是否有与本次出题相关的核心规则。

**步骤A3：读取试卷模板**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\templates\quiz-paper.md`
- 严格按照模板格式输出，不得自行创造格式。

**步骤A4：读取今日题目数据**
- 读取脚本生成的：`d:\AI\AI学习内容记忆测试\ai-quiz-system\today_quiz.json`
- 读取脚本生成的：`d:\AI\AI学习内容记忆测试\ai-quiz-system\出题指令.txt`（里面有完整的知识点核心内容）

**步骤A5：出题（🔴最高优先级：不得打乱知识点顺序！）**
- 🔴**必须严格按照出题指令中"今日待出题知识点清单"的顺序出题，不得打乱顺序、不得按题型分组、不得调整题目顺序**
- 🔴**第N题必须对应知识点清单[N]，题目编号N和知识点ID必须与清单完全一致**（违反此条会导致批改时对错更新到错误知识点！）
- 题型混合在原顺序中自然分布即可，选择题正确选项位置可以随机但题目顺序不变
- 按照知识点核心内容出题，混合选择/填空/简答题型
- 题型比例参考出题指令中的建议
- ★☆☆知识点只出选择题，不出填空/简答
- 来源标签五选一：`[薄弱点到期]`、`[未到期巩固]`、`[到期常规]`、`[新知识点]`、`[随机巩固]`
  - **薄弱点题**（薄弱点=true，即来源为`[薄弱点到期]`或`[未到期巩固]`）末尾标🔥
  - **到期常规题**（来源为`[到期常规]`）不标🔥（这类是正常到期复习，不是薄弱点）
  - `[新知识点]`和`[随机巩固]`也不标🔥

**步骤A6：强制自检卡点（⚠️ 不通过不能输出）**
- ⚠️ 必须逐项核对：`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\checklists\question-checklist.md`
- 12项全部检查通过后，才能输出最终试卷给用户。
- 特别注意第0项（顺序对应）和第1项（填空题下划线）：必须重点检查！

### 输出要求
- 开头输出试卷头（题型/来源/分级统计），来源分布必须区分薄弱点和常规到期：`🔥薄弱点题 X 题（薄弱点到期 A + 未到期巩固 B）+ 📅到期常规 C 题 + ⚪新知识点 D 题 + 🎲随机巩固 E 题`
- 题与题之间空一行
- 专业名词首次出现必须附括号中文解释
- 最后告诉用户："请按顺序作答，可以一次性写在一个回复里"

## 四、流程B：批改写回（用户答完题后）

### 执行步骤

**步骤B1：读取批改规则+经验预警**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\02-grading-rules.md`
- 重点关注：FIX#1~#7保护规则、A/B/C三种类型的更新方式、薄弱点状态机。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\risk-alerts.md`
- 批改前预警：检查是否有与本次批改相关的已知风险。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\quiz-experience.md`
- 批改时参考：检查是否有与本次批改相关的核心规则。

**步骤B2：逐题批改**
- 判断每道题对错：
  - 选择题：选对即对，选错即错
  - 填空题：核心术语意思答对即给分，不要求一字不差
  - 简答题：命中要点即给分，缺要点超过一半算错
- 错题必须写清楚：正确答案是什么、用户错在哪里、原因分析。

**步骤B3：生成grading.json**
- ⚠️ 必须严格按照模板格式：`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\templates\grading-result.json`
- 保存位置：`d:\AI\AI学习内容记忆测试\ai-quiz-system\grading.json`
- 格式：`[{"题号": N, "正确": true/false, "错因": "错误原因分析（错题必填）"}]`

**步骤B4：执行写回脚本**
- ⚠️ 强制运行脚本，不得跳过：
  ```powershell
  python d:\AI\AI学习内容记忆测试\ai-quiz-system\scripts\quiz_push.py
  ```
- 脚本自动完成：轮次更新、下次复习日期计算、薄弱点状态、写回飞书。

**步骤B5：输出批改总结+自检**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\checklists\grading-checklist.md`
- 逐项核对11项自检清单，全部通过后才能输出总结
- 按照`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\templates\grading-summary.md`格式输出
- 必须包含：正确率、薄弱点变化、今日错题、下次预告
- 如发现新问题/好做法（如新题型、新错因模式），记录后按流程E步骤E8提取经验

## 五、流程C：新知识点入库（用户发学习内容时）

### 执行步骤

**步骤C1：读取入库规则+经验预警**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\03-knowledge-ingest.md`
- 重点关注：科学性审查6项标准、查重4种策略、三问定级法。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\risk-alerts.md`
- 入库前预警：检查是否有与本次入库相关的已知风险。
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\quiz-experience.md`
- 入库时参考：检查是否有与本次入库相关的核心规则。

**步骤C2：读取分类体系**
- ⚠️ 必须读取：`d:\AI\AI学习内容记忆测试\ai-quiz-system\references\04-knowledge-taxonomy.md`
- 确定知识点归属的L0-L4层级、分配正确的ID格式。

**步骤C3：执行7步入库流程**
1. 科学性/时效性审查：检查事实错误、过时信息、翻译偏差
2. 查重去重：比对现有知识库，判断是完全重复/重合补充/交叉独立/全新
3. 结构化归属：判断属于哪个章节，还是需要新建小节/新章
4. 重要性分级：用三问判定法确定★★★/★★☆/★☆☆
5. 知识点提炼：分配ID、编写核心内容、确定标签
6. 写入飞书：通过lark-cli写入多维表格
7. 输出反馈：按照模板反馈入库结果

**步骤C4：入库反馈**
- ⚠️ 严格按照：`d:\AI\AI学习内容记忆测试\ai-quiz-system\assets\templates\ingest-feedback.md`格式输出
- 必须包含：科学性审查结果、查重结果、归属结构、分级、入库统计

### 执行步骤（必须严格按顺序执行，不得跳步）

**步骤E0-E7：修改执行（按第十章修改协议执行）**

| 步骤 | 内容 | 对应第十章步骤 |
|------|------|--------------|
| E0 | 判断变更等级（P0-P3） | 步骤0 |

**步骤E8：经验提取与沉淀（每次修改后必须执行）**
> 本次修改不管成功还是失败，都是一次经验。提取可复用的经验保存，下次碰到类似问题可迁移应用（不能生搬硬套，需根据实际情况调整）。

E8.1 经验分类（按修改结果分流）：

| 修改结果 | 提取重点 | 沉淀目标 |
|---------|---------|---------|
| ✅ 成功修复 | 有效做法、排查路径、修复模式 | 最佳实践库 best-practices.md |
| ❌ 修复失败/引入新问题 | 根因、失败路径、误判点 | 风险提示库 risk-alerts.md |
| ⚠️ 部分成功 | 有效部分+遗留问题 | 风险提示库+待验证标记 |

E8.2 结构化分析（五维度，按实际情况填写，不强求全部）：
1. **问题特征**：这次修改属于什么类型？（逻辑Bug/格式问题/流程缺陷/配置遗漏）
2. **有效做法**：什么排查方法有效？（代码追踪/对比分析/时间线推演/逐层拆解）
3. **复用条件**：这类经验在什么场景下可以迁移？（同类代码下标/同类优先级/同类触发条件）
4. **注意事项**：迁移时需要注意什么差异？（不同数组长度/不同阈值/不同执行顺序）
5. **迁移评估**：当前场景与经验原始场景的差异是什么？能否直接迁移？需要调整哪些部分？

E8.3 经验入库（按经验类型选择目标文件）：
- 写入 `references/best-practices.md`：编号 BP-XXX，包含场景描述+做法+复用条件
- 写入 `references/risk-alerts.md`：编号 RISK-XXX，包含风险描述+触发条件+规避方法
- 如果经验属于核心规则级别（如发现了新的铁律）→ 同步更新 quiz-experience.md + SKILL.md铁律 + check_consistency.py校验项
- 在 `references/CHANGELOG.md` 记录经验入库（标注 `[经验沉淀]`）
- ⚠️ 入库后运行联动检查：`python check_consistency.py linkage`，确认新经验文件有读取方

E8.4 观察期标记：
- 新沉淀的经验标注 `[观察期]`
- 经过3次同类修改验证后改为 `✅已固化`
- 验证失败的经验标注 `❌已废弃` 并说明原因

E8.5 不满意更新机制（⚠️关键）：
> 若用户对修改结果不满意（无论是立即反馈还是一段时间后在使用中发现问题），需重新修改。重新修改完成后，必须更新已沉淀的经验：

**立即反馈**（修改后用户马上表示不满意）：
- 检查E8已写入的经验条目（BP-XXX / RISK-XXX）
- 如果重新修改推翻了之前的做法 → 将原经验标注 `❌已废弃`，写明废弃原因，重新提取新经验
- 如果重新修改只是微调 → 更新原经验条目的内容，保持编号不变
- 如果重新修改发现了原经验遗漏的问题 → 补充到原经验条目的"注意事项"中

**延迟反馈**（修改后一段时间，用户在使用相关功能时才发现问题）：
- 用户触发流程A/B/C/D时反馈"之前XX修改有问题" → 判断属于哪次修改（查CHANGELOG.md记录）
- 读取该次修改沉淀的经验条目（CHANGELOG中标注了BP-XXX/RISK-XXX编号）
- 按延迟反馈处理规则更新经验：

| 延迟反馈情况 | 处理方式 |
|------------|---------|
| 修改引入了新问题（原经验说"有效"但实际有问题） | 原经验标注 `⚠️已修正`，补充问题说明和修正方案，降级为 `[观察期]` 重新验证 |
| 修改效果不持久（短期有效长期失效） | 原经验标注 `⚠️时效性限制`，补充失效条件和适用期限 |
| 遗漏了边界情况（原经验未覆盖的场景出问题了） | 补充到原经验的"注意事项"和"复用条件"中 |
| 完全推翻原经验 | 标注 `❌已废弃`，写明废弃原因，提取新经验替代 |

- 在 `references/CHANGELOG.md` 记录经验更新（标注 `[经验更新-延迟反馈]` + 原修改版本号）

|--------|------|------|-------------|

- **自指更新**：当修改的内容涉及到"修改规则本身"（即config.py/content-map.md/check_consistency.py/SKILL.md这四个三位一体核心文件）时，不仅要改业务内容，还要同步更新防护体系本身，确保护栏始终有效
- **SSOT（Single Source of Truth）**：单一事实来源，即所有常量只在config.py中定义一次，其他地方通过import引用

**违反本协议的后果**：修改一处漏改多处，导致脚本逻辑与文档规则不一致、格式错误、飞书写回失败、复习间隔计算错误；更严重的是，修改防护体系本身后不更新，会导致后续修改失去保护，问题越积越多。

### 11.5 持续迭代优化机制

- **3次修改观察期**：新经验需3次同类修改验证后正式固化（3次是平衡验证效果与迭代速度的经验值）
- **定期规则清理**：每10次修改或每月一次，检查经验库中是否有过时/冲突/低价值的经验需要清理
- **隐性经验显性化**：如果某次修改中"感觉不太对"但最终结果OK，也要记录下来分析原因，可能有隐性经验值得显性化
- **经验迁移原则**：迁移经验时必须评估当前场景与原始场景的差异，不能生搬硬套——数组长度不同则下标范围不同，阈值不同则触发条件不同，执行顺序不同则依赖关系不同
